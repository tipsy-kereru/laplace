# Freerange 활용 레시피 (Freerange Recipes)

**언어:** [English](freerange-recipes.md) | 한국어

`/laplace:freerange` (SPEC-007 설계 사양, v0.7.0 이상) 명령어를 실무에서 안전하고 효과적으로 활용하기 위한 실용 시나리오별 레시피 모음집입니다.

**시작하기에 앞서 반드시 확인해 주세요.** Freerange 모드는 수동 확인 단계를 줄여 연동 편의를 돕는 도구일 뿐이며, 외부 해킹이나 위협 명령 실행을 직접 통제하는 **보안 경계선(Security Boundary)이 아닙니다**. 개발 규율을 준수하며 호환성 있게 동작하는 정상적인 에이전트 루프는 자율적으로 매끄럽게 처리되지만, 만약 우회 작정을 하거나 악의적인 목적을 품은 AI 모델의 경우 시스템을 속이고 진행할 가능성이 있습니다. 단, 최하위 영구 차단 필터(Deny Layer - `rm -rf /`, `curl|sh`, `sudo`, `aws`, `gcloud`, `kubectl` 등)는 구조상 어떠한 경우에도 비활성화되거나 우회되지 않고 항상 강력히 강제 적용됩니다. (SPEC-007 상세 설계 사양 문서는 소스 저장소 내 [specs/](file:///home/kereru/Development/laplace/specs) 디렉토리에 보관되어 있으며 플러그인 릴리스 패키지에는 생략되어 배포됩니다.)

## 허용 스코프(Scope) 요약

| 스코프 | 수동 확인 생략(우회) 범위 | 보안 위험성 등급 |
|---|---|---|
| `flow` | 드래프트 이슈의 수동 승인 단계를 패스합니다. 프로젝트 외부망이나 시스템 환경 변수 등에는 물리적인 영향이 가지 않습니다. | 낮음(Low) |
| `publish` | `git push`, `gh pr create`, `npm publish` 등 되돌릴 수 없는 외부 형상 저장소 및 패키지 저장소 배포 작업을 패스합니다. | 중간(Medium) — 원격 서버에 코드가 영구 배포 및 게시됨 |
| `supply` | `pip install`, `npm install`, `claude mcp add` 등 신규 패키지 설치 및 툴 구성 변경 차단을 패스합니다. | 높음(High) — 에이전트 모델이 라이브러리를 직접 끌어와 자신의 실행 권한 영역(Tool Surface)을 스스로 확장 가능 |
| `all` | 위의 세 가지 영역(`flow`, `publish`, `supply`) 전체에 대한 승인 차단 게이트를 해제합니다. | 높음(High) — 영구 차단 필터만을 남겨둔 채 접수부터 릴리스까지 전 과정 완전 자율화 구동 |

`flow` 스코프가 실무에서 가장 권장되는 안전한 기본값입니다. `publish`, `supply`, `all` 모드는 승인 만료 시간(TTL)을 아주 짧게 제한하고 명확한 작업 목적이 있는 특수 시나리오에서만 신중하게 활용하세요.

## 레시피 1 — 야간 백로그 일괄 처리 (가장 권장하는 사용 방식)

**목표:** 보안 위험도가 낮은 이슈 위주로 정렬된 승인 백로그 큐를 야간 등 부재시간 동안 한 번에 처리하되, 단계마다 자리를 지키며 승인 버튼을 일일이 누르는 번거로움을 제거합니다.

**설정 절차:**
```bash
/laplace:intake docs/prd-batch.md          # 마크다운 명세를 드래프트 이슈들로 변환
/laplace:verify                            # 드래프트 이슈 요구사항 무결성 진단
/laplace:approve ISSUE-0001                # 기획 설계 적합성 확인 후 사람이 최초 1회 최종 승인
...                                        # 대상 백로그 큐들을 차례로 모두 승인 처리
/laplace:freerange on flow --ttl 8         # 8시간 유효 제한을 두고 flow 모드 활성화
/laplace:run-queue                         # 일괄 큐 기동 (에이전트가 승인 큐 순차 자동 완수)
```

**동작 및 처리 방식:** 승인 완료된 백로그 이슈들이 순서대로 추출되어 PM → Dev → Review → Security 단계를 거쳐 `review-passed` 상태까지 멈춤 없이 전이됩니다. 만약 `flow` 모드가 꺼져 있다면 매 이슈가 PM, Dev 단계 등에 진입할 때마다 승인을 위해 멈추지만, 활성화되어 있으므로 밤사이 알아서 전 과정을 처리해 둡니다. 외부 원격 push 나 신규 라이브러리 설치 시도는 강제 차단되므로 안전하며, 아침에 출근하여 바로 검수할 수 있도록 `review-passed` 결과 리포트 상태로 정렬됩니다.

**아침 출근 후 후속 조치:** `/laplace:status` 명령을 통해 밤사이 처리된 백로그들의 검증 이력을 리포트로 최종 검수합니다. 이상이 없다면 이슈별로 `/laplace:create-pr` 명령을 내려 Pull Request를 발행합니다 (외부 원격지 push 및 PR 생성은 여전히 사용자 동의를 엄격하게 대기하므로 최종 코드를 눈으로 직접 최종 검증할 기회를 확보합니다).

**왜 `all`이 아닌 `flow` 스코프를 쓰나요?** 제품 외부망으로 나가는 코드는 반드시 배포 전에 사람이 눈으로 최종 병합 diff를 직접 확인해야 하기 때문입니다. 밤사이 AI 에이전트가 단독으로 처리한 소스 코드는 '사용자가 검토할 수 있도록 준비된 상태'여야지, 임의로 원격 프로덕션 브랜치에 즉시 배포 및 반영된 상태여서는 안 되기 때문입니다.

**안전 가드레일:** 지정한 유효 한계 시간(TTL)은 다음 날 데일리 스탠드업 업무 시작 전에 만료되어 자동 비활성되므로 안심하셔도 됩니다. 상세 전환 과정은 감사 로그에 빠짐없이 기록됩니다.

## 레시피 2 — cron 기반 요구사항 수집(Intake) 및 자동 연동 (motivations 결합)

**목표:** 저장소에 신규 요구사항 마크다운(PRD) 파일이 푸시되거나 등록되면, 하네스가 이를 감지하여 드래프트 이슈로 분석 및 로드하고 큐에 배치합니다. 사람이 매번 `/laplace:intake` 명령을 타이핑하는 과정을 생략합니다.

**설정 절차:** 시스템 타이머(cron 등)를 활용해 motivations 동작과 intake 스크린 스크립트를 주기적 실행하도록 등록합니다.
```cron
# 30분 주기: motivations 트리거 실행 (승인 대기 중인 이슈 루프 재개 판단)
*/30 * * * * cd /project && python3 scripts/motivations.py --once
# 2시간 주기: 새 PRD 등록 여부를 스캔하여 드래프트 이슈로 로드 (사용자 정의 자동화 래퍼 스크립트 기동)
0 */2 * * *   cd /project && your-intake-wrapper.sh
```

래퍼 스크립트가 신규 명세를 intake하여 드래프트를 올릴 때, 수동 승인 단계를 부드럽게 패스하도록 유효 시간을 짧게 주어 flow 모드를 켜둡니다.
```bash
/laplace:freerange on flow --ttl 4
```

**동작 및 처리 방식:** 신규 명세가 업로드되면 자동화 래퍼가 이를 감지해 intake 처리하고, `flow` 스코프 활성화에 의거하여 드래프트 이슈 수동 승인 단계를 통과시키며, [scripts/motivations.py](file:///home/kereru/Development/laplace/laplace/scripts/motivations.py) 스케줄러가 백로그를 실행 큐로 밀어 넣어 `review-passed` 단계까지 완전 자동 구동시킵니다. 사용자는 다음 출근 세션에서 정제된 결과 리포트만 파악하면 됩니다.

**한계 및 주의:** flow 모드의 유효 기간(TTL)을 4시간 정도로 극히 짧게 걸어두어 주기적으로 만료되도록 하세요. `flow` 권한이 무한정으로 열려 있으면 에이전트가 통제 범위를 벗어나 원하지 않는 외부 임의 파일까지 마음대로 수집하여 자동 승인 처리할 취약점이 발생합니다.

## 레시피 3 — 신뢰 기반 배포 및 릴리스 파이프라인 (publish, 최단 시간 적용)

**목표:** 이미 모든 소스 코드 리뷰와 보안 감사를 완벽히 완수한 릴리스 이슈에 한하여, 사용자가 push 및 PR 확인 버튼을 단계별로 일일이 타격할 필요 없이, 지정한 릴리스 승인 세션 동안 즉시 일괄 배포를 지휘합니다.

**설정 절차:**
```bash
/laplace:status                           # 배포 대상 이슈가 review-passed 완료 상태인지 확인
/laplace:freerange on publish --ttl 1     # 릴리스 기동 예정 시간만큼만 (1시간) publish 모드 활성화
/laplace:release ISSUE-0042               # 원격 push -> PR 발행 -> 릴리스 일괄 자동 수행
/laplace:freerange off                    # 작업 즉시 수동으로 승인 완화 모드 종료
```

**동작 및 처리 방식:** `publish` 모드 활성화에 의해 원격 게시 시점에 작동하는 3가지 사용자 동의 수동 확인 게이트가 1시간 동안만 무력화됩니다. 에이전트 루프가 중간 멈춤 및 컨텍스트 리셋 과정 없이 신속하게 릴리스 버전 업과 push를 마칩니다.

**왜 `--ttl 1`로 극단적으로 짧게 지정하나요?** 배포는 프로젝트 형상 관리에 비가역적인(Irreversible) 영향을 미치는 고위험 행위입니다. 승인 해제 세션을 방치하지 마시고 딱 작업 기동에 소요되는 시간만큼만 열어 두어야 합니다. 또한 릴리스 성공 응답을 확인하자마자 즉시 `/laplace:freerange off` 명령어를 입력해 창을 명시적으로 차단하는 것은 배포 가이드북(런북)의 핵심 규칙입니다.

**`supply` 스코프와 절대로 동시에 열지 마세요.** 정식 버전 배포 단계에서는 어떠한 신규 라이브러리 추가나 모듈 업그레이드가 일어나서는 안 됩니다. 릴리스 빌드 도중 에이전트가 갑자기 패키지를 다운로드하려 든다면, 이는 `supply` 권한을 달라고 유도할 상황이 아니라 루프 진행에 중대한 오류가 발생했으니 즉시 멈추라는 위험 신호입니다.

## 레시피 4 — 의존성 라이브러리 일괄 업그레이드 (supply, 최고 위험 등급)

**목표:** 복수의 개발 태스크 전반에 걸쳐 요구되는 대대적인 라이브러리 업그레이드 작업(각 이슈별 `npm install` 또는 `pip install` 포함)을 순차 자동 처리할 때, 매 설치 단계마다 구동이 멈추는 간섭 현상을 배제합니다.

**설정 절차:**
```bash
# 사전 필수 준비: 업그레이드 대상 패키지는 반드시 회귀 테스트를 탑재하여 사전 검증 승인된 이슈여야 합니다.
/laplace:freerange on supply --ttl 2      # 2시간 유효 기간으로 supply 모드 활성화
/laplace:run-queue                        # 업그레이드 큐 일괄 순차 구동
/laplace:freerange off                    # 완료 즉시 수동으로 완화 모드 종료
```

**동작 및 처리 방식:** 각 개발 이슈들의 Dev 단계 기동 시, 패키지 설치 제한이 풀려 있어 멈춤 없이 라이브러리 업데이트 및 테스트 빌드를 자동 수행합니다. 단, 소스 코드 교차 리뷰(Review) 및 보안 취약점 감사(Security) 게이트는 완화 대상에 속하지 않으므로 에이전트 루프에서 여전히 꼼꼼하게 교차 필터링됩니다.

**왜 이 레시피가 가장 위험한가요?** 외부 패키지 다운로드 제한(`supply`)을 해제하는 행위는 AI 에이전트가 외부 망에서 출처를 알 수 없는 바이너리를 시스템에 직접 유치하고 도구 실행 능력(Tool Surface)을 실시간으로 무단 확장할 수 있도록 허용하는 것과 같습니다. 이는 악성 보안 위협(Supply Chain Attack 등)에 취약해집니다. 따라서 다음 완화책을 철저히 준수해야 합니다:
- 기획 이슈 명세 단계에 에이전트가 업그레이드할 대상 패키지 명과 정확한 버전 번호를 사전에 명시해 두어야 합니다.
- 승인 해제 유효 기간(TTL)은 1~2시간 이내로 한정해야 합니다.
- 작업 완료 직후 감사 로그 분석 명령(`grep '"event": "on"' .harness/logs/freerange.jsonl`)을 실행하여 의도하지 않은 무단 패키지 설치 내역이 없었는지 반드시 사람이 2차 수동 교차 검증해야 합니다.
- 본 시나리오에서는 절대 `all` 스코프를 함께 켜서는 안 됩니다. 원치 않는 오염된 패키지가 혹여나 잘못 유입되더라도 원격 저장소에 즉각 게시되지 않도록 배포(`publish`) 게이트는 무조건 묶어 두어야 합니다.

## 레시피 5 — 데모 및 샌드박스 완전 자율 테스트 (all, 휘발성 저장소 한정)

**목표:** 에이전트 하네스 루프가 사람의 개입 없이 어디까지 스스로 기능 개발과 테스트 빌드, 릴리스까지 마칠 수 있는지 실시간 데모 및 기술 실증을 하거나, 소스 코드가 손상되어도 무방한 1회성 테스트 샌드페이스를 기동할 때 사용합니다.

**설정 절차:**
```bash
/laplace:freerange on all --ttl 1         # 1시간의 타임아웃을 두고 영구 차단 필터만 가동시킨 완전 자율 모드 셋업
/laplace:pipeline                         # 명세 수집부터 빌드, 테스트, 배포 PR 발행까지 명령 하나로 일괄 구동
```

**동작 및 처리 방식:** 전체 파이프라인 시퀀스가 사람의 최종 승인 대기 없이 최종 배포 큐까지 물 흐르듯 자율 실행됩니다. 단, 이 모드에서도 `rm -rf /` 실행 방지나 시스템 환경 변수 보안키 무단 탈취 검사 등의 기본 필수 안전 규칙(Hard Safety Floor)은 내부 훅에 의해 철저하게 보호됩니다.

**반드시 격리된 샌드박스 환경에서만 사용하세요.** 프로덕션 환경의 main 브랜치나 공유 코어 브랜치 등에서 본 모드를 활성화하는 행위는 절대로 금지됩니다. 임시 테스트용 포크(Fork) 저장소나 소스 코드를 언제든 완전히 파기해도 괜찮은 독립 샌드박스 인프라 환경에서 기술 데모 용도로만 제한하여 활성화하십시오.

## 피해야 할 안티 패턴 (Anti-Patterns)

- **main(운영 배포용) 브랜치에서 `/laplace:freerange on all`을 긴 유효 기간(TTL)으로 방치하는 행위:** 장시간 동안 완전 자율 제어 모드를 활성화해 두는 것은 위험한 코드가 검증 없이 즉각 실제 사용자에게 배포되도록 방치하는 것과 같습니다. 대신 '레시피 3(publish, 짧은 배포 전용 창 활용)' 기법을 적용하세요.
- **개발 세션 종료 후에도 `supply` 권한을 계속 열어두는 행위:** 사용자가 자리를 비운 사이 에이전트가 외부 망에서 출처가 불분명한 도구를 추가로 다운로드하고 에이전트 권한 범위(Tool Surface)를 멋대로 확장할 우려가 있습니다. 해당 작업 시점에만 승인 만료 시간을 주어 구동하고, 작업 완수 즉시 명시적으로 안전하게 닫아 주세요.
- **freerange 모드를 보안 격리용 '샌드박스(Sandbox)'로 혼동하는 행위:** freerange 옵션은 사용자 개입에 따른 **'수동 승인 절차'**를 일시 생략하는 도구이지, 에이전트의 로컬 명령어 실행 범위를 물리적으로 제한하는 보안 샌드박스가 아닙니다. 만약 시스템 탈취를 작정한 악의적인 에이전트 모델이라면 로컬 권한을 오용해 보안 규칙 파일([scripts/policy.py](file:///home/kereru/Development/laplace/laplace/scripts/policy.py)) 자체를 조작할 수 있습니다. freerange 모드는 이를 물리적으로 차단하지 못합니다.
- **오류 원인을 분석하지 않고 귀찮다는 이유로 무작정 freerange 활성화하기:** 특정 패키지 설치 시 자꾸 정지 현상이 발생한다면, 게이트를 억제하기 전에 왜(어떤 이슈, 어떤 의존성)인지 알아내세요. 억제는 신호를 숨깁니다.

## 안전한 사용을 위한 관리 수칙

- **작업 세션 종료 즉시 닫기:** 기본 TTL 24시간은 비상 상황 시에 자동 해제되도록 설정된 최대 안전 상한선이지 목표치가 아닙니다. `--ttl` 옵션 파라미터를 사용해 가급적 해당 기동 예정 시간에 맞게 아주 짧게 지정하십시오.
- **감사 로그 확인 수칙:** `.harness/logs/freerange.jsonl` 파일은 추가만 가능한 안전 보존 규격입니다. `grep '"event": "on"'` 명령을 주기적으로 기동해 승인 완화 스코프와 TTL 이력이 투명하게 유지되는지 정기 감사하십시오.
- **구동 전 상태 확인 습관화:** 작업 시작 전 가볍게 `/laplace:status` 명령을 타격해 상단 대시보드에 활성화된 스코프 정보 및 잔여 유효 시간이 표시되는지 눈으로 직관적 판독을 하십시오.
- **`/laplace:freerange off` 명령어는 언제나 안전:** 본 해제 명령은 어떠한 사전 확인 및 권한 락 조건도 요구하지 않고 무조건 즉시 가동되어 보안 게이트를 정상 복구시킵니다.
